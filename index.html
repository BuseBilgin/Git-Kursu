Git nedir, Ne işe yarar? Git , versiyon kontrol sistemi olarak adlandırılan bir sistemdir. Çoğunlukla yazılım geliştirme projelerinde kullanılan bir açık kaynaklı proje yönetim sistemidir. Git, projelerde yapılan değişiklikleri takip etmek, farklı sürümler arasında geçiş yapmak ve kodları diğer geliştiricilerle paylaşmak gibi işlemleri kolaylaştırır. Versiyon kontrol istemi, hata yapıldığında projeyi önceki sürüme geri döndürmeye, zaman içinde yapılan değişiklikleri gözden geçirmeye, soruna neden olabilecek bir şeyi en son kimin değiştirdiği gibi bilgileri takip etmeye yarar. Git, ters giden bir şeyler olduğunda sorunsuz sürüme dönüş yapmayı mümkün kılar. Versiyon Kontrol Sistemlerinin Çalışma Mantığı Git'in tüm işlemlerinde ana mantık yukarıdaki resimde olduğu gibidir. Üç kısım arasındaki veri alış-verişleri sonucu proje tamamlanır. Bu kısım hakkındaki bilgiler diğer sayfalarda verilecektir. Kısacası yukarıdaki resim, kursun özetidir diyebiliriz. Git Sistemine Genel Bakış Git şeması ve Temel Kavramlar Git şemasını açıklayan bu örnek üzerinden tüm sistemi anlayabiliriz. Görsele baktığımızda işlemlerin dallara ayrıldığını ve işlemlerin birbirini takip ettiğini görebiliyoruz. 1-) İşlemin en başındaki yuvarlak head kısmını temsil eder. Adından anlaşılacağı gibi sistemin başı olan bu kısım projenin gerçekleştiği ilk aşamadır. Ardından bir sonraki işlem ile projeye devam edilir 2-) şemanın devamında bir yol ayrımı görülür. Bu kısma ise dal manasına gelen branch denir. Branch oluşturmadaki temel amaç, projeye iki farklı koldan giderek devam etmek ve isteğe göre bunları birleştirmektir. 3-) Şemanın devamında projenin tekrardan kollara ayrıldığı görülmektedir. Bunların birleşme (merge) ve ayrışma işlemlerini komutla gerçekleştirmek mümkündür. Kursun devamında buna değinilecektir. Git sistemindeki veri alışverişi yandaki şekiller arasında olur. Özelliklerini tanımlayacak olursak: Working directory: Bu kısım kodların yazıldığı mevcut çalışma alanıdır Staging area: Şekilde görüldüğü üzere ortada kalan bu kısım, aktarılma aşamasında olan dosyanın nereye gideceğine karar verilen kısımdır. Repository: Bu kısım, dosyaların saklandığı final kısmıdır. Eklenen değişiklikler projeye bu aşamada tamamen aktarılır. Bu bölümlerin işlevlerine ileride detaylı olarak değineceğiz. Komutlara Giriş: Dosya İşlemleri Git'in kullanımı için terminal veya komut satırı üzerinden çeşitli komutlar kullanılır. Bu komutlar sayesinde dosyaların takibi yapılabilir, değişiklikler kaydedilebilir ve projenin farklı versiyonları arasında geçiş yapılabilir. Git'i kullanmak istiyorsanız, ilk olarak terminal veya komut satırına giriş yaparak Git komutlarına hakim olmanız gerekmektedir. Bu bölümde komut satırlarına giriş yapacağız. 1) pwd Komutu pwd (present working directory) komutu, kullanıcının bulunduğu anlık dizin yolu bilgisini terminale yazdıran bir komuttur. Yani hangi dizinde olduğunuzu gösteren bir komuttur. Komutu çalıştırdığınızda, terminalde şu an nerede olduğunuzu görebilirsiniz. 2) ls Komutu ls Bu komut, bulunduğunuz dizindeki dosya ve klasörleri listeler ve bunların isimlerini terminale yazar. "ls" komutu, seçenekler ve argümanlarla birlikte kullanılabilir ve bu şekilde farklı çıktılar üretebilir. Örneğin, "-l" seçeneği kullanıldığında dosya ve klasörlerin ayrıntılı listesi görüntülenir. Elbette bu dosyaları listelemek istememizin bir nedeni var. Bu dosyalar arasında gezinerek hedef dosyaya geçiş yapmamız gerekiyor. Bu düşünceyle kursa devam edebiliriz. 3) clear Komutu clear Dosyalar arasında geçiş yapmadan önce terminal ortamını temizlemek gerekebilir. Bu komut, kullanıcının önceki komutları ve çıktıları temizlemesine izin verir ve ekranda daha net bir görünüm sağlar. Örneğin, "clear" komutunu kullanarak bir önceki konsol oturumunuzun çıktılarını ve komut geçmişini temizleyebilirsiniz. 4) cd Komutu cd "cd" komutu, "change directory" yani "dizin değiştir" anlamına gelir. Bu komut, kullanıcının bulunduğu dizini değiştirmesine izin verir. Komutun kullanımı şu şekildedir: Örneğin, "cd /home/kullanici/adresler" komutu, kullanıcının "/home/kullanici/adresler" dizinine gitmesini sağlar. 5) mkdir Komutu mkdir "mkdir" komutu, "make directory" kelimelerinin kısaltmasıdır ve yeni bir dizin (klasör) oluşturmak için kullanılır. Bu komutu kullanarak, mevcut dizinde veya başka bir dizinde yeni bir alt dizin oluşturabilirsiniz. Komutun kullanımı şu şekildedir: mkdir [dizin_adı] Burada [dizin_adı], oluşturulacak yeni dizinin adını belirtir. Örneğin, "mkdir yeni_dizin" komutu, "yeni_dizin" adlı yeni bir dizin oluşturur. 6) touch Komutu touch Bu komut belirtilen dosyaların var olmasını sağlar veya varolan dosyaların değiştirilme tarihini günceller. Eğer touch komutuna varolmayan bir dosya adı verilirse, o isimde bir dosya oluşturulur. Eğer varolan bir dosya adı verilirse, o dosyanın son erişim ve değiştirilme tarihi güncellenir. Ayrıca, touch komutu ile dosya içeriği değiştirilmez, sadece dosyanın meta verileri değişir. Kısaca özetlemek gerekirse, touch komutu dosya oluşturmak veya dosyanın son değiştirildiği zamanı güncellemek için kullanılır. Böylelikle dosya işlemlerini bitirmiş olduk. Bu komutlar git ile doğrudan bağlantılı olmasa da dosyalar arasında geçiş yapmak projeyi yönetmek açısından olmazsa olmazdır. Temel dosya komutlarını öğrendiğimize göre git komutlarına geçiş yapabiliriz. Git Komutlarına Giriş : Git init ve Git status Artık Git' in kendi komutları üzerinden işlem yapabilecek seviyeye geldik. Bu kısım ve bundan sonraki bölümlerde Git komut ekranı üzerinden oluşturduğumuz proje üzerinden işlemler gerçekleştireceğiz. Başlatma, ekleme, birleştirme ve geri dönüş işlemleri dahil olmak üzere pek çok komut satırıyla ilgileneceğiz. Git init ve Git status Repo ile ilgili işlemlerde sıklıkla bu iki komutu kullanacağız. Git init ve git status komutları birbiriyle ilintili olduğundan dolayı ikisini birlikte ele alacağız. Git init: git init komutu, bir dizinde yeni bir Git deposu (repository) oluşturmak için kullanılır. Bu komutu çalıştırdığınız klasör artık bir git deposu olarak ayarlanır ve bu depoda bulunan dosyaların sürüm kontrolü yapılabilir hale gelir. Git status: git status komutu, çalışılan Git deposundaki dosyaların durumunu görüntülemek için kullanılır. Bu komutla, hangi dosyaların takip edildiği, hangilerinin eklenmediği, hangilerinin değiştirildiği ve hangilerinin silindiği gibi bilgiler elde edilebilir. Aynı zamanda, git status komutu ile bir sonraki adımda hangi komutların kullanılabileceği hakkında da ipuçları alınabilir. Bu komut, Git deposunda yapılan değişikliklerin durumunu kontrol etmek için sıkça kullanılır. yukarıdaki örneğe bakıldığında git status komutuyla kontrol etmek için öncelikle git init komutu çalıştırılmalıdır. git init komutu için ise önceki bölümde öğrenmiş olduğumu cd komutu işe yaramaktadır. Özeytleyecek olursak öncelikle giti başlatmak için bir klasör seçilmelidir. Hangi dizinde oluşturulmak isteniyorsa o dizine girilmelidir. Daha sonra git init ile git başlatılır. ardından git status ile dosyalar kontrol edilir. Dosyaları, oluşturduğumuz git sistemine nasıl aktaracağımızı sonraki aşamalarda öğreneceğiz. Git Add ve Commit İşlemi Git init ile mevcut dizinde git başlatma işlemini öğrenmiştik. Şimdi ise git add ve git commit komularıyla istediğimiz dosyaları repoya eklemeyi öğreneceğiz. Önce git add ve git commit komutlarının işlevini inceleyecek, daha sonra bunları önceki bilgilerden hareketle bir örnekle pekiştireceğiz. Yandaki şekli daha önce "Git Sistemine Genel Bakış" sayfasında görmüştük git add komutu, git init ile başlattığımız git sistemine dosya eklememizi sağlar. git add komutu ise söz konusu içeriği, çalıştığımız dizinden bekleme kısmı olarak tabir edebileceğimiz bölüme aktarmaya yarar. örnek kullanım : git add ornek_dosya Bu kullanımla birlikte ornek_dosya isimli dosyamızı staging area kısmına aktarmış olduk. Bu dosyayı dilersek repostiyory kısmına aktarabiliriz. Bu işlem için de git commit komutuna ihtiyaç duyarız. Yine isteğe bağlı olarak commit işlemine mesaj eklenebilir. Eklenen mesaj sayesinde yapılan değişikliğin neleri içerdiği diğer ekip üyeleri tarafından görülebilir. örnek kullanım : git commit -m "sifre.txt dosyasina yeni sifreler eklendi" Öğrendiklerimizden Hareketle Add ve Commit İşlemlerini Gerçekleştirelim : İlk olarak hangi dizinde olduğumuzu anlamalıyız. Son olarak tüm dosyaları staging areadan repository kısmına taşıyacağız. Bunu yaparken git commit komutunu kullandık. "Dosyalar eklendi" bilgisiyle commite ait bir mesaj bıraktık Git Log ve Gitignore Git add ve git commit işlemlerini önceki bölümde öğrenmiştik. Bu bölümde ise genel olarak commit işleminden sonraki süreçleri takip edeceğiz. Gitlog Nedir , Gitlog Nasıl Kullanılır? Bir veya birden fazla kez commit işlemi yaptığımızı farz ediyoruz. Bunların bilgilerine erişmek için git log komutunu kullanmamız gerekiyor. Komutu çalıştırmadan önce git logun ne olduğuna bir göz atalım: git log komutu, bir Git deposundaki commit geçmişini görüntülemek için kullanılır. Bu komut, en son yapılan commit'ten başlayarak tüm commitleri listeler ve her bir commit'in SHA-1 karma değeri, yazarı, tarihi ve mesajını gösterir. Ayrıca, belirli bir düzenleme ya da sorgulama yapmak için farklı seçeneklerle de kullanılabilir. Bazı sık kullanılan parametreler şunlardır: --oneline: Her bir commit'i tek bir satırda özetler. --graph: Commit geçmişini grafiksel olarak gösterir. --author: Sadece belirli bir yazarın commit'lerini listeler. --since veya --until: Belirli bir tarih aralığındaki commit'leri listeler. Örneğin, git log --oneline komutu, tüm commit'leri tek bir satırda özetleyecektir. Commit işleminden sonra her commite bir adres atanır. Bu adreslerin nasıl kullanılacağı konusunda daha sonra bilgi sahibi olacağız Gitignore : Bazı Dosyaların Görünmesini Engelleyebilirsiniz Gitignore dosyası, Git'in belirli dosyaları veya dizinleri takip etmemesini sağlayan bir dosyadır. Bu, proje klasörünüzdeki özel veya hassas dosyaların yanı sıra gereksiz veya geçici dosyaların yüklenmesini ve paylaşılmasını engellemek için kullanılabilir. Bir gitignore dosyası oluşturmak için, proje kök dizininize .gitignore adında bir dosya eklemeniz gerekir. Bu dosya, takip edilmemesi gereken dosya ve dizinlerin listesini içerecektir. Dosya isimleri, dosya uzantıları ve dizin adları gibi şablonlar kullanarak belirli dosyaları veya türleri hariç tutabilirsiniz. Örneğin, bir Node.js projesinde "node_modules" klasörü genellikle gereksizdir ve depolama alanı kaplayabilir, bu nedenle .gitignore dosyasına "node_modules/" ekleyerek bu klasörün Git tarafından takip edilmemesini sağlayabilirsiniz. .gitignore dosyasının içeriği aşağıdaki gibi görünebilir: # .log uzantısına sahip dosyaların görünmesini engellemek için kullanabilirsiniz *.log # "temp" isimli dizinin içindeki her şeyi gizler temp/ # yalnızca "config.json" ismindeki dosyayı gizler config.json Yukarıda görüldüğü gibi dosyaları gizlemenin farklı yolları var. Bu dosyaların görünmesini engellemek için .gitignore dosyasına başvurduk. Ancak yukarıdaki kısım tam anlaşılmamış olabilir. Bu yüzden öğrendiklerimizi terminalde uygulayarak daha iyi anlayabiliriz. İster terminalden touch .gitignore komutuyla ister klasörün içinde yeni dosya oluşturarak gitignore dosyasını oluşturabiliriz. Buradaki hedefimiz .idea dosyasının projemizde görünmemesini sağlamak. git init ile önceden projeye git sistemini entegre etmiştik. Şimdi git status komutunu kullandığımızda oluşturduğumuz .gitignore dosyasının gite eklenmediğini görüyoruz. bunu git status ile anlıyoruz. git add .gitignore komutuyla ignore dosyamızı gite ekliyoruz. Böylelikle ignore dosyamızda yaptığımız değişiklikleri versiyon kontrolüne tâbi tutmamız artık mümkün. .gitignore dosyamıza görünmesini istemediğimiz dosyaların listesini ekliyoruz. Bunları herhangi bir kod editörü aracılığıyla ekleyebiliriz. Yazdıklarımızı kaydediyoruz. Gitignore dosyalarına eklenmesi gereken içeriklerin neler olması gerektiğine dair bazı sitelerde bilgiler mevcut. Örneğin c dilinde bir proje dilinde geliştirdiyseniz ve gitignore dosyanızda nelerin yer alacağını bilmiyorsanız c gitignore file template olarak aratabilir, örnek şablon dosyasını projenize ekleyebilirsiniz. git status komutunu kullandığınızda .gitignore dosyasına yazıp kaydettiğiniz içeriklerin otomatik kaydedildiğini göreceksiniz. Böylelikle projenizde hangi dosyaların görünmesini istemediğinizi belirtmiş olacaksınız. GİT BRANCH NEDİR BU BRANCHLER VE NE İŞE YARARLAR? Git, dağıtık bir versiyon kontrol sistemi olarak kullanılır. Git branch(dal) projenizin farklı versiyonlarını kullanmanızı sağlayan bir kavramdır. Branch oluşturmaktaki esas amacımız projemize farklı kollardan giderek devam etmek ve isteğe bağlı şekilde bunları birleştirmektir. Git branchler, aşağıdaki çeşitli senaryolarda kullanılabilir: Yeni özellik eklemek Hataları düzeltmek Farklı denemeler yapmak Her dal, ana(master) dalından ayrı olarak takip edilen bir geliştirme yolunu temsil eder. Kısacası, git branch, projenizde farklı yönlerde çalışmanıza olanak tanıyan ve kod tabanını izole edebileceğiniz bir yapıdır. Bu sayede bir dalda yapılan değişiklikler, ana dalı etkilemeden geliştirilebilir ve yönetilebilir. Git branch ile birlikte aşina olmamız, bilmemiz gereken farklı kavramlarda mevcuttur. Bu kavramlardan kısaca bahsedelim. HEAD NEDİR? Çalışma alanınızdaki mevcut konumu temsil eden bir işaretçidir. HEAD, en son yapılan işlemi veya son commiti gösterir. MERGE NEDİR? Merge işlemi, farklı geliştiricilerin aynı projede ayrı ayrı çalıştığı durumlarda veya yeni özelliklerin ana dal ile birleştirilmesi gerektiğinde sıklıkla kullanılır. Bu şekilde, projedeki farklı geliştirme dalları birleştirilerek tek bir ortak kod tabanı elde edilir. Merge işlemini pratikte komutlarla daha iyi anlamak için algoritmik bir haritayla açıklayalım: git branch < dal-adı > komutu Git'te yeni bir dal (branch) oluşturmak için kullanılan bir komuttur. Yeni bir dal oluşturmak için 'git branch' komutunu kullanırken, oluşturmak istediğiniz dalın adını komutun sonuna eklemeniz gerekmektedir. Örneğin, git branch feat komutunu çalıştırarak "feat" adında bir dal oluşturabilirsiniz. Bununla birlikte, oluşturduğunuz yeni dal, mevcut konumda (HEAD) kalır. git commit komutu Üstteki şeklimizde gördüğümüz üzere ' feat' adında yeni bir branch oluşturduk. Fakat yukarıda branchte hiçbir işlem gerçekleştirmedik. git commit komutunu kullanarak, Git deposunda yapmış olduğunuz değişiklikleri kalıcı olarak kaydedebiliriz. Yani Oluşturduğunuz yeni dalda gerçekleştirdiğiniz değişiklikleri commit olarak kaydetmek için bu komutu kullanabilirsiniz. Yandaki şekilde de takibini sağladığımız gibi isteğimize bağlı git commit komutunu kullanarak bir commit daha atabiliriz. git switch < dal-adı > git switch komutu, Git deposundaki çalışma alanınızı belirtilen dala değiştirmek için kullanılan bir komuttur. Kaba şekilde bahsedersek yandaki şekilde de gösterildiği gibi git switch master komutuyla master dalında çalışma yapabiliriz. Sonrasında masterda , git commit komutuyla işlemler gerçekleştirebilir, örnek haritamızda yandaki şekildeki gibi bir görüntü ile bağdaştırabiliriz. git merge < kaynak-dal > git merge komutu, Git'te iki farklı dalı birleştirmek için kullanılan bir komuttur. Bu komut, değişiklikleri kaynak dalı hedef dala uygulayarak dalları birleştirir. Örneğin, yandaki görselimizde şematize edildiği gibi git merge feat komutunu kullanarak 'feat' dalını 'master' dalına birleştirebiliriz. FAST FORWARD NEDİR? Örneğin 'feat' adında bir branch açalım ve üzerinde işlemler gerçekleştirelim. 'master' dalında hiçbir şey gerçekleştirmemiş olalım ve sonrasında bu brancleri birleştirelim (merge). Bu işleme FAST FORWARD adı verilir. İşlemler aslında tek dal üzerinden de yürütülebilir görünüyor. Neden fast forwarda ihtiyaç duyulur? Birden fazla kişi çalışıyor olabilir. Deneme yapmak için ayrı branch açıp eski kodları bozmamak için bu işlem gerçekleştirilebilir. Peki her iki branchte yapılan işler birleştirme esnasında çakışırsa (conflict) ne olur? Buna 'Merge Conflict' (Birleştirme çakışması) adı verilir. MERGE CONFLİCT Git'te iki veya daha fazla dalın birleştirilirken aynı dosyadaki farklı değişikliklerin çakışması durumunda ortaya çıkar. Bu çakışma, otomatik olarak çözülemeyen ve manuel müdahale gerektiren durumdur. Merge conflict'i çözmek için genel adımlar: Merge conflict durumunu tespit edin: git status komutunu kullanarak merge conflict durumunda olan dosyaları belirleyebilirsiniz. Bu dosyalar genellikle çakışma göstergeleriyle (<<<<<<<, =======, >>>>>>>) işaretlenir. Çakışan dosyaları düzenleyin Düzenlenen dosyaları kaydedin: Dosyaları düzenledikten sonra, değişiklikleri kaydedin (commit). Git, bir versiyon kontrol sistemi olduğu için projelerinizde geriye dönük işlemler yapmanızı kolaylaştırır. 'checkout' komutu, repository'in farklı dallarına ya da önceki işlem geçmişine gitmenizi sağlar. Önceki konulardan öğrendiğimiz gibi git, her bir commit için farklı bir kod (hash) oluşturur. Bu hash kodları, checkout komutu ile geçmişte belli bir noktaya gitmek için kullanılabilir. Örneğin, "git checkout 123abc" komutunu girerek, 123abc kodlu bir commit'e geri dönebilirsiniz. Ayrıca, checkout komutu ayrıca farklı branch'lara geçiş yapmanızı da sağlar. Yeni bir branch oluşturduktan sonra, "git checkout" komutunu kullanarak o branch'a geçebilirsiniz. Aynı şekilde, daha önce oluşturulmuş bir branch'a da geçiş yapabilirsiniz. Bir diğer kullanım alanı ise geçici değişiklikler yapmak istediğiniz zamanlardır. Örneğin, bir dosyada değişiklik yapmadan önce o dosyanın son halini görmek istiyorsanız, "git checkout" komutunu kullanarak son commit haline geri dönebilirsiniz. Bu sayede dosyalarınızda değişiklik yapmadan önce, son halini görme şansına sahip olursunuz. Birkaç örnek üzerinden 'git checkout' komutunu irdelemeye devam edelim: 1. Örnek: En son commit haline geri dönmek için git checkout -- Bu komut, üzerinde çalıştığınız dosyaların son halini geri getirmek için kullanılır. '--' işareti, son commit haline gitmek istediğinizi belirtir. 2. Örnek: Başka bir branch'taki değişiklikleri almak için git checkout my-branch Bu komut, 'my-branch' adlı başka bir branch'taki değişiklikleri almak için kullanılır. Bu komuttan sonra üzerinde çalıştığınız dosyalar, 'my-branch' branch'ındaki dosyaların en son haline göre güncellenir. 3. Örnek: Geriye doğru belli bir commit'e gitmek için git checkout 123abc Bu komut, '123abc' kodlu commit'e geri dönmek için kullanılır. Bu komuttan sonra üzerinde çalıştığınız dosyalar, '123abc' commit'e göre güncellenir. 4. Örnek: Bir dosyanın en son halini geri getirmek için git checkout -- filename.txt Bu komut, 'filename.txt' adlı dosyanın en son commit haline geri getirir. Dosya adını doğru girdiğinizden emin olun. Tüm bunların yanında yukarıda gördüğümüz kodlardan bazılarını terminal üzerinde denemek konuyu daha iyi kavratacaktır. Önceki konulardan hareketle branchlerin nasıl oluşturulacağına dair bilgi sahibiyiz. Birden fazla branchimiz var ve geçiş yapmak için önce branchlerimiz listelememiz gerekiyor. "dev" isimli branche geçiş yapmak istiyoruz. Böylelikle tüm dosyalar dev branchinin son haline göre güncellenmiş oldu Yine önceki bilgilerden hareketle git log komutundan faydalandığımızı varsayıyoruz. git log sayesinde commitlere hash kodu atandığını öğrenmiştik. Şimdi bunun üzerinden küçük bir işlem yapacağız. git log sayesinde atanan hash kodları aracılğıyla istediğimiz bir sürüme dönüş yapacağız. Bunun için git log listesindeki hashlerden istediğimizi kopyaladığımızı düşünüyoruz Ardından git checkout hash_id formatını kullanarak istediğimiz sürüme geçiş yapıyoruz 'detached HEAD' kısmında bir uyarı dikkatimizi çekiyor. Checkout işleminden sonra head konumumuzun değiştiğini görüyoruz. Başka bir deyişle geçiş yapmış olmamız, HEAD değiştirdiğimiz anlamına geliyor. Revert İşlemi Revert İşlemi Git revert, bir önceki "commit" işlemini geri almaya yarayan bir git komutudur. Bu sayede, son yapılan değişikliklerin tersine çevrilmesi sağlanır. Git revert komutunu kullanarak, belirli bir commit veya bir commit aralığı geri alınabilir. Komut çalıştırıldığında, seçilen commit veya commit aralığındaki değişikliklerin tersini yapacak yeni bir commit oluşturulur ve bu commit git geçmişinde görünür hale gelir. Git revert komutu, bir değişiklik tarihçesinde geri almak istediğiniz herhangi bir değişikliği geri alma imkanını sunar. Bu, yanlışlıkla yapılan veya uygunsuz bir şekilde kaydedilmiş bir değişikliği geri almanıza olanak tanır. Git revert komutunu kullanmak için şu adımları izleyebilirsiniz: 1. Git çalışma dizininize gidin. 2. `git log` komutunu çalıştırarak geri almak istediğiniz commit’in SHA numarasını bulun. 3. `git revert ` komutunu kullanarak geri almak istediğiniz commit’i geri alın. Bu işlem bir revert commiti oluşturacaktır. 4. Eğer birden fazla commit'i geri almak istiyorsanız, `git revert ..` komutunu kullanarak start ve end olarak geri almak istediğiniz commit aralığının SHA numaralarını belirtin. Örneğin, `git revert HEAD` komutu son yapılan commit'in tüm değişikliklerini geri alacaktır. `git revert HEAD~2..HEAD` komutu ise son iki commit arasındaki değişiklikleri geri alacaktır. Geri alma işleminden sonra, değişiklikleri kontrol etmek ve doğru bir şekilde geri alınıp alınmadığını görmek için `git log` komutunu kullanabilirsiniz. Geri alma işleminden sonra, bu yeni commit'i git sunucusuna veya paylaşılan depoya yükleyebilirsiniz. Yine alıştırmalar üzerinden ilerlemeye devam edelim: git log komutu ile commitlerimizi kontrol ediyoruz. Buradakilerin tamamı örnektir. Commit oluşturma ve log adımlarını önceki sayfalarda öğrenmiştik. örnek olarak üç adet dosya oluşturalım ve içlerine rastgele metinler girelim commit işleminden sonra git log ile kontrol edelim dosya eklendi isimli commiti geri almak istiyoruz. Bunun sonucunda dosya klasörden silinsin ancak log listesinde görünmeye devam etsin istiyoruz. İlgili hash kodunu terminale yapıştırarak komutu giriyoruz git log komutunu çalıştırdıktan sonra son commit işleminin geri alındığı bilgisiyle karşılaşıyoruz Klasöre geri döndüğümüzde revert işlemini yaptığımız dosyanın silindiğini görüyoruz. Böylelikle revert işlemini bitirmiş oluyoruz. Hem dosyanın silinmesi hem de commit listesinden silinmesi için revert işlemi yerine reverse işlemini kullanmamız gerekiyor. Reset İşlemi Reset İşlemi Git reset komutu projedeki değişiklikleri geri almak için kullanılır. Bu komutun kullanımı oldukça esnektir ve farklı parametreler alabilir. Temel olarak, git reset komutu ile son commit işlemi geri alınarak projeniz önceki bir duruma getirilir. Bu durumda, son commit ile proje arasındaki tüm değişiklikler kaybolur. Aşağıdaki adımlarla git reset komutunu nasıl kullanabileceğinizi açıklayabilirim: git log komutuyla geri almak istediğiniz commit'in kimliğini (hash değerini) bulun. git reset komutunu kullanarak geri almak istediğiniz commit'e göre çalışma dizininizi güncelleyin. Örneğin, son iki commit'i geri almak için git reset HEAD~2 komutunu kullanabilirsiniz. Geri alma işlemi tamamlandıktan sonra, dosyalarınızı tekrar düzenleyebilir, eklemeler yapabilir ve yeni bir commit oluşturabilirsiniz. git reset komutunun alabileceği parametreler şunlardır: --soft: Geri alma işlemi yapılırken Git sadece commit'i geri alır ve çalışma dizinindeki dosyalara dokunmaz. Dosyalarınızın son hali üzerinde çalışabilirsiniz ve yeni bir commit oluşturabilirsiniz. --mixed: Bu, varsayılan moddur ve git reset komutu bu parametre ile çalıştırıldığında Git, commit'i geri alır ve çalışma dizininizi günceller, ancak değişiklikleri bir sonraki commit'e hazır hale getirmez. Yani dosyalarınızın son hali üzerinde çalışabilirsiniz, ancak yeni bir commit oluşturmadan önce yeniden git add komutunu kullanarak değişikliklerinizi eklemeniz gerekiyor. --hard: Bu parametre, git reset komutunun en keskin modudur ve geri alma işlemi yapılırken tüm değişikliklerinizi siler. Böylece çalışma dizininiz, geri alınan commit'in tam haliyle eşleşir. Bu nedenle, bu modda çalışırken dikkatli olmanız gerekiyor. Özetle, git reset komutu Git'in güçlü araçlarından biridir ve projenizde geri alma işlemleri yapmak için kullanılabilir. Ancak, tüm değişikliklerinizi kaybetmek istemiyorsanız, doğru parametreyi seçtiğinizden emin olmalısınız. Dikkatli olunması gerektiği için git reset --hard komutunu ele alacağız: Revert işleminde yaptıklarımızın aynısını yapıyoruz. git log komutu ile commitlerimizi kontrol ediyoruz. Buradakilerin tamamı örnektir. Commit oluşturma ve log adımlarını önceki sayfalarda öğrenmiştik. örnek olarak bir adet klasör ve iki adet dosya oluşturalım ve git klasörünü commitleyelim commit işleminden sonra git log ile kontrol edelim dosya oluşturuldu isimli commiti geri almak istiyoruz. Bunun sonucunda dosya klasörden silinsin ve resetleyeceğimiz log listeden silinsin. git log komutunu çalıştırdıktan sonra "dosya oluşturuldu" isimli commitin silindiğini görüyoruz. Klasöre geri döndüğümüzde reset işlemini yaptığımız dosyanın silindiğini görüyoruz. Böylelikle reset işlemini bitirmiş oluyoruz. Böylelikle git reset işlemini tamamlamış oluyoruz. GitHub GitHub Nedir ? GitHub, yazılım geliştiricilerin kodlarını saklamak, paylaşmak ve işbirliği yapmak için kullandıkları popüler bir sürüm kontrol platformudur. Projelerin değişikliklerini izlemek ve yönetmek için kullanılır. Aynı zamanda açık kaynak projelerin geliştirilmesine olanak sağlar. GitHub Profiliniz GitHub profiliniz, dijital kimliğiniz ve yazılım geliştirme yeteneklerinizi sergileyebileceğiniz bir platformdur. Profiliniz, projelerinizi, katkıda bulunduğunuz açık kaynak projeleri ve kişisel bilgilerinizi içerebilir. GitHub profilinizdeki projeler, diğer geliştiricilerin ilgisini çekebilir ve işverenler veya işbirliği yapmak isteyen kişiler tarafından fark edilebilir. Ayrıca profilinizdeki etkinlikler, paylaştığınız kodları ve projelerdeki katkıları göstererek tecrübenizi ve işbirliği becerilerinizi kanıtlar. İyi bir GitHub profilinin, yazılım geliştirme topluluğunda tanınmanıza, yeni fırsatlar elde etmenize ve işbirliği yapabileceğiniz kişilerle bağlantı kurmanıza yardımcı olabileceği unutulmamalıdır. GitHub Explore GitHub Explore, GitHub platformunda keşif yapmak ve ilginizi çekebilecek projeleri, toplulukları ve kullanıcıları bulmak için kullanılan bir özelliktir. GitHub'ın sağladığı bir dizi arama ve filtreleme seçeneğiyle keşfetmek istediğiniz konuları veya projeleri kolayca bulabilirsiniz. GitHub Explore, açık kaynak projelerden trendlere, popüler repolardan geliştirici profillerine kadar geniş bir yelpazede içeriği sunar. Yenilikçi projeleri, ilham verici kod tabanlarını ve yeni teknolojileri keşfetmek için ideal bir araçtır. Bu özellik sayesinde, farklı programlama dilleri, konular veya kategoriye göre filtreleme yapabilir ve ilgi alanlarınıza uygun projeleri bulabilirsiniz. Ayrıca, diğer geliştiricilerin paylaşımlarını inceleyerek topluluk içindeki etkinlikleri takip edebilir ve ilgilendiğiniz projelere katkıda bulunabilirsiniz. GitHub'da repository açmak GitHub'da bir depo (repository) oluşturmak, projelerinizi yönetmek, kodlarınızı paylaşmak ve işbirliği yapmak için önemli bir adımdır. Depo oluşturma işlemi oldukça basit ve hızlıdır. İlk adım olarak, GitHub web sitesine gidin ve hesabınıza giriş yapın. Ardından, sol üst köşedeki yeşil "New" simgesine tıklayın. Repository Name (Depo Adı): Depo için bir isim belirleyin. İsim, projenizin neyle ilgili olduğunu ve neyi içerdiğini anlatmalıdır. Description (Açıklama): Projenizin kısa bir açıklamasını ekleyin. Bu, diğer geliştiricilerin projenizi daha iyi anlamalarına yardımcı olur. Public/Private (Halka açık/Özel): Depoyu halka açık mı yoksa özel mi yapmak istediğinizi seçin. Halka açık depolar herkese görünür ve katkıda bulunulabilirken, özel depolar sadece sizin veya belirli bir ekip üyelerinin erişebileceği şekilde olur. Initialize this repository with a README (Bu depoyu README dosyasıyla başlat): Bu seçeneği işaretlerseniz, depo ile birlikte otomatik olarak bir README dosyası oluşturulur. README, projenizin açıklamasını, kurulum talimatlarını, kullanım kılavuzunu vb. içerebilir. Add .gitignore: Bu seçeneği kullanarak, projenizdeki belirli dosyaların veya dizinlerin Git tarafından izlenmemesini sağlayabilirsiniz. Örneğin, derleme dosyalarını veya gizli verileri .gitignore dosyasıyla belirtebilirsiniz. Choose a license (Lisans seçin): Projenize bir lisans ekleyebilirsiniz. Lisans, projenizin kullanım koşullarını belirtir ve diğer geliştiricilerin projenizi kullanmalarını düzenler. Bu bilgileri doldurduktan sonra, "Create repository" (Depo oluştur) düğmesine tıklayarak depoyu oluşturabilirsiniz. Depo oluşturulduktan sonra, Git komutlarını kullanarak kodlarınızı depoya ekleyebilir veya doğrudan GitHub arayüzünden dosyaları yükleyebiliriz. Repository Komutları Git Push "Git push", yerelde yaptığınız değişiklikleri remote repository'e göndermek için kullanılan bir Git komutudur. Bu komutu kullanarak, yerelde yaptığınız commit'leri remote repository'ye yükleyebilirsiniz. Diğer ekip üyeleri bu değişiklikleri görüntüleyebilir, değerlendirebilir ve projeye katkıda bulunabili Git Pull "Git pull", remote repository'deki değişiklikleri yerel depoya çekmek için kullanılan bir Git komutudur. Yani, başkalarının yaptığı güncellemeleri alarak yerel projenizi güncel tutabilirsiniz. Git pull komutu, "git fetch" ve "git merge" komutlarını birleştirir. Fetch işlemi ile remote repository'deki değişiklikler yerel depoya getirilir, ardından merge işlemi ile bu değişiklikler yerel projenize entegre edilir. Git Fetch "Git fetch", remote repository'deki değişiklikleri yerel depoya getirmek için kullanılan bir Git komutudur. Ancak, fetch komutu, yerel projenizdeki değişiklikleri otomatik olarak birleştirmez. Yani, yereldeki projenizde herhangi bir değişiklik yapmadan önce, remote repository'deki değişiklikleri önce "git fetch" komutu ile getirir ve ardından bu değişiklikleri yerel projenize birleştirme işlemini yapabilirsiniz. Git Clone "Clone", bir remote repository'nin tam bir kopyasını yerel bir bilgisayara almak için kullanılan bir Git komutudur. Clone işlemi, bir projeyi başlatmak veya mevcut bir projeyi yerelde çalışmak için kullanışlıdır. Clone işlemi ile remote repository'nin tüm geçmişi, dosyaları ve dalları yerel depoya indirilir. Bu sayede projeyi yerelde düzenleyebilir, değişiklikleri yapabilir ve ardından push ve pull gibi komutlarla remote repository ile senkronize edebilirsiniz Git Fork "Fork", bir projenin kopyasını oluşturmak için kullanılan bir işlemdir. Özellikle açık kaynak projelerde yaygın olarak kullanılır. Bir projeyi fork ettiğinizde, projenin tam bir kopyasını kendi GitHub hesabınıza alırsınız. Bu sayede projeyi değiştirebilir, yeni özellikler ekleyebilir ve bunları remote repository'ye geri gönderebilirsiniz. Fork işlemi, projeye katkıda bulunma imkanı sağlar ve orijinal projenin etkilenmeden kalmasını sağlar. Projeyi fork etmek için, GitHub sayfasında sağ üst köşede bulunan "Fork" düğmesine tıklayın. Bu, projenin tam bir kopyasını kendi GitHub hesabınıza almanızı sağlar. Fork işlemi tamamlandıktan sonra, fork ettiğiniz projenin GitHub deposu artık sizin GitHub hesabınızda bulunur. Bu depoyu yerel bilgisayarınıza klonlamak için aşağıdaki komutu kullanabilirsiniz: Artık fork edilen projeyi yerelde düzenleyebilirsiniz. Değişiklikler yapabilir, yeni özellikler ekleyebilir veya hataları düzeltebilirsiniz.
